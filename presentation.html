Why this matters to NOC

You are the ones who see it first

Part 1 · Concept

What is a WAF?

  • Sits in front of the web app — first line of defense.
  • Works at the application layer (HTTP/HTTPS).
  • Inspects each request — URL, headers, body — and blocks malicious ones.
  • Catches what a traditional network firewall can't see: SQLi, XSS, and friends.

⏱ ~2 min
Keep it short — most of the deck is demos. Key contrast: a network firewall filters by IP/port (layers 3–4); it happily passes a valid HTTP request that contains an injection. A WAF reads the actual HTTP content (layer 7) and understands attack patterns. Analogy: network firewall = the building's door lock; WAF = the security guard reading what's in the envelope.

Part 1 · Attack 1 live demo

SQL Injection — login bypass

The login builds its query by gluing your input into SQL text:

SELECT * FROM users
WHERE user = 'admin' --' AND pass = ''

The -- comments out the password check. You're in as admin, no password.

Demo steps

  1. Go to /login.html
  2. Username: admin' --
  3. Password: leave blank → Sign in
  4. Show the returned executedSql
⏱ ~3 min · DEMO (presenter on screen — participants watch)
Do it live. First log in normally as guest/guest to show it works. Then log in with admin' -- and blank password → you're admin. Point at the executedSql field in the response so they see how the input rewrote the query. Teaching point: the DB did exactly what it was told — the bug is trusting user input as code. Mention the real fix is parameterized queries, but the edge fix is DP WAF spotting the injection pattern. Note: bare ' OR '1'='1 won't work here (the AND still applies) — you need the -- comment. Good moment to explain precedence if the room is technical.

Part 2 · Concept

What is load balancing?

Spreading incoming traffic across multiple origin servers so no single one is overwhelmed.

With DP WAF you don't run your own — you give it your origin URLs and it balances for you.

Algorithms:

⏱ ~2 min
Quick concept. Explain the three algorithms in one line each: Weighted RR — bigger servers get more; Least Conn — send to the least busy; IP Hash — same client always lands on the same origin (sticky sessions). We'll use IP Hash in the demo. Reinforce DP WAF's model from the product: "you give us origin URLs, we balance" — no extra infra for the team to run.

Part 2 · Concept

DDoS protection — per-IP awareness

  • A DDoS floods the service from many machines (a botnet) to exhaust resources.
  • DP WAF runs a log analyzer watching for flood patterns per IP, continuously.
  • Abusive IPs go on a block list; FortiGate drops their traffic before it reaches the origin.
  • Load balancing spreads legitimate load; per-IP blocking removes the abusive load.

This is exactly the IP-awareness you saw in the capacity demo, applied defensively.

⏱ ~2 min
Connect the dots: the same "count by IP" logic that limited capacity is what powers DDoS defense — identify the noisy IPs, drop them. Two layers working together: load balancer spreads the good traffic, per-IP block list + FortiGate removes the bad. For NOC: this is what turns a "site down under attack" page into a non-event. Mention what they'd see on dashboards: a spike from a narrow set of IPs that then flatlines once blocked.
Wrap-up

Takeaways for NOC

  • 🛡️ WAF = layer-7 guard. Blocks SQLi & XSS — injections the network firewall can't see.
  • ⚖️ Load balancing = availability. Horizontal scale removes the single-origin wall.
  • 🚧 Per-IP control = DDoS defense. Spread the good, drop the bad, at the edge.
  • 👀 You're the first responders. Now you know the signatures: 503 spikes, single-IP floods, injection strings in logs.
⏱ ~2 min
Land the plane. Repeat the one-liner: the edge (WAF + LB) is where both security and availability are enforced, before the origin. Encourage them to keep the lab and replay it. Invite questions before the quiz.
